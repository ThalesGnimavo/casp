# Changelog

## 0.13.0 — 2026-07-21

A new **drift category** for the half of the protocol that had no gate behind it: the queue. Everything before this validated the *head* of the queue (`state.next_prompt`) and the integrity of what had already *shipped* — nothing validated the ordering of what had not run yet. `next_after` was a template convention with no rule behind it. Four codes now prove the plan itself is executable. Additive and backward-compatible: no existing finding changes verdict, the `check --json` schema stays **v1**, no new state key, and every prior test stays green (121 → 143).

- **New — `CASP-PROMPT-007` … `CASP-PROMPT-010`, prompt-chain integrity.** The documented workflow is to draft a run of prompts ahead of time, chain them with `next_after`, point `next_prompt` at the head, and execute one per session without re-planning. That workflow is only as good as the chain, and the chain could be silently incoherent while `casp check` stayed green. Now: `007` — a `next_after` naming a slice that resolves to nothing (**FAIL**); `008` — a cycle, including a prompt naming itself (**FAIL**); `009` — two queued prompts claiming the same predecessor (**WARN**); `010` — a queued prompt no chain from the head reaches (**WARN**).
- **The severity split is deliberate.** A dangling reference and a cycle are claims that *cannot be true* — the plan is unexecutable as written, so they gate the push. A fork or an orphan is *ambiguous* rather than false, and a deliberate parking lot of queued-but-unchained prompts is a legitimate way to work; reddening it would punish a real workflow.
- **It cannot redden a repo that has not opted in.** The canonical template ships `next_after: <previous-session-id-or-prompt-slug>` as a literal placeholder, so a large share of real prompts in the wild carry an unedited value. A placeholder, an empty string and `null` are **not declarations** and produce nothing. A repo where no queued prompt declares a real predecessor gets **no finding at all** — silence, the same treatment `CASP-SESSION-003` gives a repo where no log declares a phase. Adoption is derived from the data: no new state key, nothing to configure. Prompts skipped as non-declaring are counted in the PASS line, never silently.
- **Resolution is exact, never fuzzy.** A `next_after` resolves against evidence the validator already reads: a prompt in the sessions directory (by filename stem, its lowercase form, or its slug with the scaffold's `PHASE-` prefix removed), a session id that maps to a log (the `CASP-SESSION-001` resolver), or a phase id declared by state. Every match is an exact string after a documented normalization — `CASP-SESSION-003` refused fuzzy filename matching and so does this. A near miss is a dangling reference, not a resolved one.
- **Identity collisions resolve by rule, not by `readdir` order.** `A.md` and `PHASE-A.md` both answer to the slug `a`. Identities are ranked most-specific-first (stem, lowercase stem, slug) and inserted least-specific-first over a path-sorted list, so an exact stem always wins and the resolver is a pure function of the filenames — the same repo cannot resolve differently on two filesystems.
- **History is not blamed.** Only *queued* prompts are subjects. A shipped prompt is a valid resolution target — a chain legitimately terminates on the slice that ran before it — but its own `next_after` is never re-litigated.
- **No orphan storm on top of a real finding.** When `next_prompt` is missing or already shipped, `CASP-PROMPT-001` / `003` already FAIL; the reachability walk stays silent there rather than burying the one actionable finding under a warning per queued prompt.
- **`casp status --json` gains `queue`** — the resolved chain as repo-relative prompt paths, head first, so an agent can plan more than one session ahead. `null` when the chain is not adopted or not coherent (no guessed order). Additive: the status schema stays **v1**, and `status` still exits `0` on a drifted chain — reporting, never gating.
- **FIX (pre-existing, found while adversarially probing this slice) — a non-string `next_prompt` crashed the whole gate.** `state.json` accepts any JSON, and `next_prompt: 42` (or a list, or an object) reached `join()` and threw `ERR_INVALID_ARG_TYPE`, killing the entire report with a raw stack trace instead of emitting a finding. A crashed gate is worse than a red one: CI reads it as infrastructure failure rather than as drift, and every other check in the run is lost with it. A non-path value is now a `CASP-PROMPT-001` FAIL naming the offending type. Same class as the scalar-`state.json` crash fixed in 0.12.1; `casp next` and `casp status` were already guarded, `check` was not.
- **Twenty-five new regression tests** (146 total), each pinning a reproduced case rather than the abstract rule: a never-adopting repo emits not even a PASS line / an unedited placeholder, `null` and `''` produce nothing / no non-string YAML value is ever treated as a declaration (number, float, boolean, list, map, `~`, whitespace-only) / a hostile-shaped `state.json` reports drift instead of crashing, for every container type / a coherent chain PASSes and states its skipped count / the slug, session-id and phase-id forms each resolve / a dangling reference FAILs and exits 1 / a near-miss filename is dangling, not fuzzy-matched / a two-node cycle is one finding, not one per node / a self-reference FAILs / a fork WARNs without blocking / an orphan WARNs without blocking / a parked prompt is never an orphan / a shipped prompt's dangling `next_after` is ignored while the prompt itself stays a valid target / a missing head produces no orphan storm / all four codes appear in `casp rules` and resolve in `casp explain` / every chain finding carries a rule code / `status --json` exposes the order, and `null` on a drifted or unadopted chain.
- **Found by dogfooding, and it worked.** On its first real fixture — this repository — the rule immediately surfaced that the queue was not actually chained: two queued prompts named slices that had shipped long ago, so the plan's real order was ambiguous. The frontmatter is fixed in this release and the chain now resolves.

## 0.12.1 — 2026-07-21

**Fixes two data-loss paths in `casp upgrade` (0.12.0). Upgrade immediately; 0.12.0 is deprecated on npm.** Found by an adversarial review that reproduced both by execution, after 0.12.0 had already been published — the publish was premature and the fix is same-day.

- **FIX — a symlinked parent directory let a write escape the cockpit.** The symlink check inspected only the final path component, but every parent is followed by `mkdirSync`/`writeFileSync`. A symlinked `casp/templates/` — or a symlinked `casp/` itself, ordinary in a monorepo — meant `upgrade` wrote into, and overwrote files in, a directory it was never pointed at. No race was required; a plain run did it. Every write target's resolved parent must now sit inside the resolved cockpit, and an unresolvable path fails closed.
- **FIX — the refresh set was a denylist, so unknown files were overwritten by default.** It named `state.json`, `now.md` and `roadmap.md` and refreshed everything else the package ships. Any root-level template added in a future release would therefore have silently replaced whatever the operator had written at that path, with no warning line. It is now an **allowlist**: `README.md` plus everything under `casp/templates/`. A shipped file this code has not heard of is skipped, not written — the default for the unknown is "leave it alone".
- **FIX — an unrecognised flag was ignored, so a mistyped dry run performed a real write.** `casp upgrade --dryrun` (and `-N`) applied. On the one verb that writes into a user's repository, unrecognised now exits 2 without writing anything.
- **FIX — a `state.json` holding a bare scalar crashed the run.** `42` parses, is truthy and is not null, so it passed the guard and the property assignment threw under strict mode — after the scaffolds were written and before the version stamp, leaving a raw stack trace. A state file that is not an object is now reported as a broken cockpit and left alone.
- **FIX — `saveState` is atomic.** It was a naked truncating `writeFileSync`: a crash or a full disk mid-write destroyed the state file it was updating. Now a temp file plus `rename`, which is atomic within a filesystem. Pre-existing and shared, so this also hardens `ship`, `close` and `audit`.
- **Four new regression tests** (121 total), each pinning a reproduced exploit rather than the abstract rule: an escaping symlinked parent leaves the outside file byte-identical and creates nothing there; an unrecognised root-level file survives an upgrade; `--dryrun` / `-N` / `--force` each exit 2 and write nothing; a scalar `state.json` produces no `TypeError`.

## 0.12.0 — 2026-07-21

The protocol's own continuity across releases. `casp upgrade` closes the gap where CASP could improve its scaffolds but had no way to deliver them to a repo already on CASP. One new command makes it a minor bump; everything is additive — no existing finding changes verdict, `casp check` gains no rule, the `check --json` schema stays v1, and every prior test stays green (118 total).

- **New — `casp upgrade`, refresh a cockpit's scaffolds without eating its state.** Before this verb, `init` refused on an existing `casp/` and `init --force` overwrote *every* file the package ships under `templates/` — `state.json`, `now.md` and `roadmap.md` included. The only refresh path destroyed the operator's data. `upgrade` is the surgical alternative: it rewrites **only** `casp/README.md` and `casp/templates/**`, and it **never writes `now.md` or `roadmap.md` and never templates over `state.json`**. The single state write is additive — stamp `casp_version`, round-tripped through the parsed object so every existing value stays byte-identical. It never deletes (a scaffold the package stopped shipping is left alone), it is idempotent (a re-run on a current cockpit writes nothing and says so), and `--dry-run` / `-n` prints the same plan and writes nothing. No `casp/` → exit 1 pointing at `init`. `upgrade` never gates; `check` remains the only gate.
- **Why now.** It was a no-op by accident from 0.7.0 to 0.9.0, when zero scaffold bytes changed between releases. **0.11.0 ended that**: it shipped a new session-log template carrying the `phase:` frontmatter that `CASP-SESSION-003` reads — so the newest rule was unadoptable by every cockpit scaffolded before it, short of copying the file by hand or losing the state file. The refresh list is *derived* from what the package ships minus an explicit data denylist, so the next scaffold change is delivered without touching this code.
- **New (additive) — `casp_version` in `state.json`.** The CASP release that last scaffolded or upgraded the cockpit. `casp init` stamps it; `casp upgrade` refreshes it. **Optional and never gating**: a cockpit scaffolded before the stamp existed carries none and still PASSes `casp check` — asserted by a regression test, because a new required key would have reddened every existing user repo on upgrade. Documented in `schemas/state.schema.json`.
- **`casp doctor` — a cockpit-staleness `WARN`.** A new `cockpit.version` probe compares the stamp against the installed CLI: `PASS` when equal, `WARN` when the cockpit is older (pointing at `casp upgrade`), when it carries no stamp at all, or — the inverse anomaly — when it was stamped by a *newer* CASP than the one installed. `WARN`, never `FAIL`: doctor still always exits `0`.
- **Namespaced on purpose.** The stamp is `casp_version`, not `version`. `state.json` has always accepted arbitrary extra keys, so a bare `version` would have silently overwritten a project recording its *own* product version — in the one verb whose entire promise is that no existing value changes.
- **Hostile filesystem shapes are handled, not crashed on.** A **symlink** in the cockpit points somewhere the operator chose, possibly outside `casp/`; `upgrade` reports it as `symlink` and never writes through it. A **write that fails** (a directory in the way → `EISDIR`, an unwritable file → `EACCES`, a regular file where `casp/templates/` belongs → `ENOTDIR`) is reported per file and the run continues to the version stamp and to exit `0` — an abort there would have left the cockpit half-refreshed *and* unstamped, so every re-run would repeat the same partial write. A broken install with no packaged `templates/` says so instead of throwing an `ENOENT` stack.
- **`casp/README.md` is a scaffold, and the output says so.** It is refreshable by design, but it is also the one refreshable file an operator plausibly hand-edits, so its `refresh` line carries `(replaces your local edits — keep notes in now.md)`. The closing summary vouches only for the three data files, deliberately.
- **The scaffolded-on date is preserved.** `casp/README.md` records the date the cockpit was created — a fact about that repo, not a template value. `upgrade` reads the existing date back out and keeps it, so a README that differs from the package copy only by its date reports `same` and a run today and a run next month agree.
- **Also — a coverage test that was assumed to exist and did not.** Nothing asserted that the verbs `src/help.ts` documents and the verbs `src/cli.ts` dispatches are the same set; the hardcoded list in `test/help.test.mjs` had fallen five verbs behind without failing. The invariant is now checked structurally in both directions, so a verb can never ship dispatched-but-undocumented or documented-but-undispatched.
- **Sixteen new regression tests** (118 total): a stale cockpit is refreshed while `now.md` / `roadmap.md` / every non-stamp state value stay byte-identical / the README date survives a refresh / a second run writes nothing / `--dry-run` and `-n` write nothing and do not move the stamp / a missing scaffold directory is re-added with the 0.11.0 template / a symlinked scaffold is left alone and its out-of-cockpit target is never rewritten / a directory in the way is reported per file while the run still stamps and exits 0 / refreshing `README.md` warns that local edits are replaced / `README.md` is pinned as the only shipped non-data scaffold carrying `{{TODAY}}` (a tripwire: a future one would break idempotency silently) / no `casp/` exits 1 with the init pointer / `init` stamps the version and the fresh cockpit still checks green / an unstamped cockpit still PASSes `check` / doctor's four version verdicts, all exiting 0 / `upgrade` is a first-class verb in both the top-level deck and its own help block / an unrecognised flag never turns `upgrade` destructive / help ↔ dispatch coverage in both directions.

## 0.11.0 — 2026-07-21

First new **drift category** since the rule codes landed: `CASP-SESSION-003` ties the shipped scoreboard to the written record. A new rule can change a repo's verdict, so the whole design is built so it cannot redden a repo that has not opted in — every prior test stays green (102 total), the `check --json` schema stays v1, and no existing finding changes verdict.

- **New — `CASP-SESSION-003`, every shipped phase must be declared by a session log.** A `phases_shipped[]` entry is a claim that a slice shipped; this rule requires a session log to say so. Severity **FAIL** with a `→ fix` hint, so it gates the push like any other drift.
- **The mapping is declared, never inferred.** A log opts in with a `phase:` key in its YAML frontmatter — a scalar id, or a list when one session shipped several. **Filenames are never consulted**: a log named `26-07-19-001-0-10-0-audit-watermark.md` against the phase id `0.10.0-audit-watermark` would need exactly the fuzzy match this category refuses to make. If the mapping would need a guess, there is no finding.
- **Adoption is derived from the data, not configured.** No new state key. `phases_shipped` is ordered and append-only, so the first entry any log declares opens the enforcement window; every entry from there on must be declared, and everything before it is exempt as pre-adoption (the exempt count is printed, never silent). A repo where no log declares a phase gets **no finding at all** — silence, the same treatment migrations get in a project that tracks none. Backfilling an old log only moves the window earlier; it never fabricates history.
- **Retroactive adoption, without lying.** A repo adopting CASP with years of history behind it is not asked to invent logs it never wrote. Two honest paths: declare from the next session on (history stays exempt), or backfill `phase:` into the old logs that do exist — which pulls the window earlier and enforces everything after it. Removing an entry from `phases_shipped` is the third: a scoreboard that claims less is not drift.
- **Template — `casp new log` now scaffolds `phase:`** in the session-log frontmatter, with the placeholder documented inline. An unedited placeholder declares a phase that `phases_shipped` does not contain, so the window stays closed and the check stays silent until adoption actually anchors.
- **Scope, stated honestly.** This catches *the scoreboard claiming more than the record supports*. It does **not** catch shipping without recording — a release that never gets appended to `phases_shipped` leaves nothing for this rule to test. That gap is the queued `facts-layer` phase's job, not this one's.
- **Ten new regression tests** (102 total): never-adopted repo stays silent / fully declared passes / an undeclared shipped phase FAILs and exits 1 / retroactive adoption exempts pre-adoption history and names the window / a gap after adoption still FAILs while history is not retroactively blamed / one log declaring a list satisfies each entry / a declared phase absent from `phases_shipped` keeps the window closed (guards the `findIndex` → `slice(-1)` trap) / a suggestively-named log without `phase:` does not count / a directory named `*.md` in the logs dir is skipped instead of crashing the report with `EISDIR` / non-string `phase:` values (number, bool, null, nested map) are ignored rather than declared.

## 0.10.0 — 2026-07-19

Two-tier verification pass. One new command (`audit`) makes it a minor bump; everything is additive and backward-compatible — no existing finding changes verdict, `casp check` does not gain a rule, the `check --json` schema stays v1, and every prior test stays green (92 total).

- **New — `casp audit`, the deep-audit watermark.** CASP now names the boundary between two verification tiers that were being conflated. The **cheap per-merge tier** — `fmt` + typecheck + lint + the touched module's own tests — is gated by `casp check` and runs every session; it is the only thing standing between a refactor and an irreversible bug (money, cross-tenant). The **expensive holistic tier** — an adversarial sub-agent audit + the full e2e battery + a security review — is slow and token-heavy, so it runs in **batch, on demand, over everything merged since the last one**, never per session. `casp audit status` prints the unaudited range `last_deep_audit..HEAD` (commit + file counts, `--json` for data); `casp audit bump [<sha>]` records HEAD (or a given commit) as deep-audited. A new optional `last_deep_audit` state field (short SHA) holds the watermark. **It is a production-cutover gate, never a merge gate: `casp check` does not block on it** — a stale watermark delays a deploy, it never blocks a commit. `bump` resolves its ref through injection-safe git (`gitArgs`) and refuses a non-commit; `status` treats a rebased-away watermark as orphaned and asks for a re-baseline. Managed end-to-end by the new `/audit-batch` skill (`skills/audit-batch`), which scopes the review from `audit status`, runs the battery + auditor once over the range diff, and bumps the watermark only on GO.
- **Motivation.** Running the holistic pass every session was turning a ~7-minute close into ~40 minutes of largely duplicated or batchable work (a full local `verify` CI already re-runs, a token-heavy sub-agent audit, a full e2e battery). Separating the tiers keeps the irreversible-bug gate on every merge while letting the semantic/security/e2e risk accumulate safely behind an explicit, queryable watermark that must clear before a cutover.
- **Schema — `last_deep_audit` documented** in `schemas/state.schema.json` (optional; a project that never runs the batch pass never sets it). Fully additive.
- **Five new regression tests** (92 total): status with no watermark (whole tree unaudited) / bump writes HEAD and status then reports up to date / status counts commits + files since the watermark / bump on a bogus ref exits 1 and leaves state untouched / unknown subcommand exits 1.

## 0.9.0 — 2026-07-15

DX + machine-handoff pass. Two new commands (`doctor`, `version`) make it a minor bump; everything is additive and backward-compatible — no existing finding changes verdict, the `check --json` schema stays v1, and every prior test stays green.

- **New — `casp doctor`, a read-only environment diagnostic.** Answers "is this machine set up to run casp?" — not "has the state drifted?" (that stays `casp check`'s exclusive job). It probes the **environment**: Node version (>= 20), the `git` binary and repository, `casp/state.json` presence + validity, the resolved `sessions_dir` / `logs_dir`, `core.hooksPath`, and whether a CASP-managed pre-push hook is installed. Prints `PASS` / `WARN` / `FAIL` per line, plus `--json` (own `schema_version`). **It never gates — always exits `0`, even on a `FAIL`**: a map of what to fix, not a gate. It reuses the same directory resolver and hook-detection as the rest of the binary, so its verdicts never diverge from `check` / `install-hook`. Documented in [docs/doctor.md](docs/doctor.md).
- **New — `casp version` + `casp version --json`.** The plain form prints the version (byte-identical to `casp -V` / `casp --version`, which are unchanged). `--json` emits the machine handoff `{ name, version, node, schema_version }`, where `schema_version` is the `check --json` report schema version — so an agent can negotiate the check-report shape from one call. `name` and `version` are read from `package.json` (single source of truth, via a shared reader).
- **New (additive) — structured `expected` / `actual` on `check --json` findings.** Where a finding has a single natural expected-vs-actual pair — `last_commit.git` (HEAD vs the recorded sha), `next_prompt.status` (`queued` vs `shipped`), `migrations.match` (on-disk vs in-state) — the JSON finding now carries `expected` / `actual` so a consumer can diff without regex-parsing `detail`. Both are `null` on every other finding. **Fully additive: `schema_version` stays `1`**, no field renamed or removed, finding order unchanged, and the human report is byte-identical (the fields are JSON-only). The published `check-result.schema.json` and [docs/check-json.md](docs/check-json.md) are updated in lockstep.
- Thirteen new regression tests (87 total): doctor healthy / missing-state (FAIL but exit 0) / invalid-JSON / `--json` envelope + summary tally / hook WARN→PASS after `install-hook` / outside-a-repo WARN; `version` plain == `-V` and `--json` shape with matching `schema_version`; `expected` / `actual` populated on the shipped-prompt and unresolvable-`last_commit` drifts and `null` on passes.

## 0.8.0 — 2026-07-14

Protocol-maturity + hardening pass. New commands (`rules`, `explain`) make it a minor bump; everything else is additive and backward-compatible — no existing finding changes verdict, the `check --json` schema stays v1, and every prior test stays green.

- **New — stable rule codes on every finding, plus `casp rules` / `casp explain`.** Each `casp check` finding now carries a public `CASP-<AREA>-<NNN>` code (e.g. `CASP-GIT-001`, `CASP-PROMPT-003`). `casp check --json` gains an **additive** `rule` field (no `schema_version` bump — still v1); `casp check` prints the code next to each WARN/FAIL. `casp rules` lists the catalogue (`--json` for data); `casp explain <CODE>` prints one rule's definition — what it verifies, the evidence it inspects, remediation — and also resolves an internal finding id. The registry (`src/rules.ts`) is the single source; a coverage test fails if any emitted finding id maps to no rule. Codes are a compatibility surface: they change only through explicit deprecation, never silently. Internal dot-namespaced `id` values are unchanged.
- **Security — injection-safe git for untrusted values.** `git()` is shell-based and must only take static, literal arguments; a new `gitArgs()` passes an argv array to `execFileSync` with **no shell**. Every call that interpolates untrusted input — `state.last_commit`, `sessions_dir` / `logs_dir`, CLI refs (`verify`, `state diff`) — now routes through `gitArgs()`, so a crafted value like `HEAD; rm -rf ~` becomes one invalid git argument (git errors → the check FAILs), never a shell command. Regression test included. The remaining literal-only `git()` calls carry no injection surface; migrating them is tracked defense-in-depth. New `docs/threat-model.md` records the trust boundaries.
- **New — published JSON Schemas.** `schemas/state.schema.json` and `schemas/check-result.schema.json` (Draft 2020-12), now shipped in the npm package. Structural conformance tests assert `casp init` produces every schema-required state key and `casp check --json` matches the result shape — so the schemas can't drift from the binary.
- **Docs — precise claim scoping.** New `docs/what-casp-proves.md` states exactly what a clean `casp check` proves (recorded execution state consistent with git evidence) and what it does **not** (code correctness, deployment, remote DB migrations, business intent). New `docs/rules.md` catalogues the rule codes. README updated: rule codes in the sample output, `rules`/`explain` in the command deck, links to the scope and threat-model docs.
- **Fixes (previously queued):** the installed pre-push hook runs under `set -eu` (POSIX; `pipefail` intentionally omitted as a `#!/bin/sh` bashism); `casp state diff` array deltas are now multiset-correct (dropping one of a duplicated element reports a single removal instead of cancelling out — a display-only path, never gating); a new advisory `CASP-MIGRATION-003` WARN fires when a configured `migrations_dir` holds migration files but `migrations_applied` is unset (silent for a genuinely empty dir).
- Eleven new regression tests (74 total): injection blocked; multiset delta; migrations-untracked WARN + empty-dir silence; `rules`/`explain` behavior; rule-coverage on clean and drifted states; both JSON schemas' structural conformance.

## 0.6.0 — 2026-06-17

> **Behavior change — `casp next` now gates on drift.** `casp next` runs the validator before printing and **refuses to emit the next prompt when the state has drifted** (exit 1, drift summary on stderr, nothing on stdout). This is intentional: it closes the *start* boundary the same way `install-hook` closes the *push* boundary. If you script `casp next` and rely on it always printing, add `--no-check` to restore the old behavior. Both boundaries are now gated — you can neither start nor finish a session on a lying state.

- **New — `casp next` pre-session gate.** Before printing the next prompt, `casp next` runs the same in-process validator as `casp check` (`checkOne` — it never shells out to itself) and refuses on any FAIL: drift summary to stderr, stdout stays empty, exit 1. The missing-file and already-shipped cases keep their existing sharper messages; the gate catches the rest (stale `last_commit`, unverifiable claims, …). `--no-check` is the explicit escape hatch (you own the drift); `--no-git` skips the git-dependent checks. `next` stays a printer — it never runs anything after printing (anti-roadmap). Four new tests: clean prints, drift refuses with empty stdout, `--no-check` bypasses, missing-prompt keeps its specific guard.

- **New — `casp verify <commit>` + `casp state diff`.** Two read-only inspection verbs that make "git log is your compliance trail" actionable. `casp verify <commit>` materializes a commit in a throwaway detached worktree, runs the exact same validator there, propagates the verdict, and ALWAYS removes the worktree (the exit code is computed inside the try and applied after the finally, since `process.exit()` skips finally blocks — exiting mid-try would leak the worktree). `casp state diff [A] [B]` reports a field-level diff of `casp/state.json` between two commits (default `HEAD~1` → `HEAD`), with element-level deltas for array fields like `phases_shipped`; human output, or `--json`. Both never mutate the worktree, index, or history. Five new tests: state diff array/field deltas + no-op, verify clean (exit 0, no leaked worktree), verify a historically drifted commit (exit 1, still cleans up), verify a non-commit (exit 1).

- **New — `casp status --json`.** A machine-readable continuity snapshot: current/next phase, next prompt + its status, last commit/session, shipped/queued counts, git head/branch/ahead, and an **embedded validator verdict** computed in-process (the same `checkOne` `check` runs). Stable schema documented in [docs/status-json.md](docs/status-json.md) with the same stability promise as check-json. Crucially, `status --json` **never gates** — a valid cockpit always exits `0`, even when the embedded verdict is `drift` (reporting is not gating; that is `check`'s and `next`'s job). The structured session handoff, and the unit a future `status --all` fleet roll-up would aggregate (sketched, not built). Three new tests: shape + clean verdict, drift embedded but exit 0, human output intact.

- **New — `casp install-hook`.** One verb writes an executable `.git/hooks/pre-push` that runs `casp check --quiet`, turning "check before every push" from discipline into mechanism. This is the install-hook tier-1 item: a hands-off self-verifying loop auto-commits and pushes — it will not run `casp check` by hand — so wiring the deterministic state gate into `pre-push` is what makes CASP fire *inside* the autonomous loop instead of being a step the agent skips. The harness self-verifies that the *code* works; `casp check` on every push self-verifies that the *recorded state* still matches git, the one check the loop structurally can't do for itself.
- **Safe by construction.** The hook carries a `CASP-MANAGED-HOOK` marker. Installing refuses to clobber a pre-existing foreign `pre-push` hook unless `--force`; re-installing over CASP's own hook is idempotent (refresh, exit 0). `casp install-hook --remove` uninstalls only a hook CASP wrote, and refuses a foreign one. The hook resolution prefers `npx --no-install @justethales/casp` (the locally-installed package, no network) and falls back to a `casp` on PATH — probing with `--version` first so "package not resolvable" never gets mistaken for the check's own drift verdict.
- **Boundaries held.** Explicit opt-in only — `casp init` never installs the hook. CASP never writes `core.hooksPath` or any git config; when `core.hooksPath` is set (git ignores `.git/hooks` then), the verb refuses with a clear message rather than writing a silently-dead hook. Worktrees and `.git`-as-a-file resolve correctly (the path comes from `git rev-parse --git-path`).
- Eight new regression tests (42 total): hook installed + executable + marked; idempotent re-install; the installed hook run directly **blocks a drifted push (exit 1)** and **passes a clean one (exit 0)** via the documented `casp`-on-PATH fallback; foreign-hook refusal + `--force` override; `--remove` removes ours and refuses a stranger; `core.hooksPath` set → refuse without touching the config; linked-worktree resolution to the common hooks dir; outside a git repo → exit 1.

## 0.5.0 — 2026-06-16

- **New — configurable `sessions_dir` / `logs_dir`.** Two OPTIONAL `state.json` keys let a project point the validator at its real layout instead of adopting CASP's. Prompts still default to `docs/plan/sessions` and logs to `session-logs`; set either key and the *entire* protocol honors it — `casp check` (every claim, the shipped-history dirs, the state-bump surface, the uncommitted-changes pathspec), `casp new prompt|log`, `casp ship` (the wired `session_log` pointer), and `casp close` (newest-log detection). Modeling a rolling file is a different concept and was deliberately not built.

## 0.4.0 — 2026-06-15

- **New — `casp ship <slug>`.** The mechanical half of closing a session, automated: flips the phase's prompt to `status: shipped`, wires its `session_log` pointer, and moves the slug from `phases_queued` to `phases_shipped`. Resolves the slug against `docs/plan/sessions/*.md` by normalized name (so `0.4-close-loop` matches `PHASE-04-CLOSE-LOOP.md`), or takes `--prompt <path>`. Idempotent on re-run; refuses an unknown slug or a `pending` session-log id. **Touches no git** — the operator owns the commit.
- **New — `casp close`.** The guided deterministic close: auto-detects `last_commit` (HEAD) and `last_session_id` (newest session log), lets you confirm or override them (`--yes` / `--commit` / `--log` for non-interactive use), bumps `updated_at`, then runs `check` and exits with its verdict. `last_commit` is set to the current HEAD so the operator's own state-bump commit lands as the canonical close-loop PASS. **Never runs git** — no add, commit, or push. A state verb, not a harness.
- **New — `casp check --all [root]`.** Validates every `casp/` cockpit under a root in one report (per-cockpit verdict + aggregate), exits 1 if any cockpit drifts. `--all --json` emits a per-cockpit array under the existing schema. The fleet gate for "many agents, many repos." Internally, per-root validation was extracted into a pure `checkOne(root)` — single-root output and exit code are unchanged.
- **Change — migrations are now fully opt-in.** A project with no `migrations_dir` and no `migrations_applied` gets **no migration finding at all** (not even a PASS), so non-code cockpits (content, launch ops, research) carry zero migration noise. `casp init` no longer scaffolds `migrations_dir: "drizzle"`. A non-empty `migrations_applied` with no `migrations_dir` set is drift (FAIL): a claim with nothing to verify against. Existing repos that declare `migrations_dir` are unaffected.
- Ten new regression tests (27 total) covering ship, close (including the never-commits invariant), optional-migration silence, and `--all` aggregation.

## 0.3.0 — 2026-06-10

- **Fix (verdict-changing) — no more false green when a claimed directory is missing.** A check that cannot find what it needs no longer silently skips: when `state.json` makes a claim that requires a directory and that directory is absent, `casp check` now **FAILs** with a `cannot verify <claim>` finding. Three claims are enforced: a real `last_session_id` requires `session-logs/` (`last_session.logs_dir`), a non-empty `migrations_applied` requires the migrations directory (`migrations.dir` — the canonical drift example itself was a false green when the dir was missing), and a non-empty `phases_shipped` requires both `docs/plan/sessions/` and `session-logs/` (`shipped_history.*`). Any other commit past `last_commit` stays WARN. `casp check` on CASP's own repo is now fully green (13 PASS · 0 WARN · 0 FAIL), previously a permanent explainable WARN.
