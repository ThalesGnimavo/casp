# 26-06-17-002 — 0.6.0: gate both boundaries + inspection

Bundles the rest of 0.6.0 on top of `install-hook` (`26-06-17-001`): the
pre-session gate, the structured snapshot, and the two inspection verbs.

## What shipped

- **`casp next` pre-session gate.** Before printing, `next` runs the validator
  in-process (`checkOne` — never shells out to itself) and refuses on any FAIL:
  drift summary to stderr, stdout empty, exit 1. Closes the *start* boundary
  symmetrically with `install-hook`'s *push* boundary. `--no-check` waives it,
  `--no-git` skips git checks. Behavior change → prominent CHANGELOG warning.
- **`casp status --json`.** Machine-readable snapshot (phase/next/last, shipped &
  queued counts, git head/branch/ahead) plus an embedded in-process check
  verdict. Never gates: a valid cockpit always exits 0, even on drift. Stable
  schema in `docs/status-json.md`.
- **`casp verify <commit>`.** Runs the validator against a historical commit in a
  throwaway detached worktree, propagates the verdict, and always removes the
  worktree (exit code computed in `try`, applied after `finally`, since
  `process.exit()` skips finally blocks).
- **`casp state diff [A] [B]`.** Field-level diff of `casp/state.json` between two
  commits (default `HEAD~1` → `HEAD`), element-level deltas for array fields;
  human or `--json`. Read-only.

## Scope

- `install-hook`, `next`, `status --json`, `verify` / `state diff` are additive
  tooling — they do not change what `casp check` *means*. `check-shipped-log`
  (a new drift category, verdict-changing) is deferred to its own slice so it
  cannot silently redden repos that carry pre-adoption shipped phases.

## Verbs canon

- Core verbs unchanged (`init` / `status` / `check` / `next` / `new`). The 0.6
  additions (`install-hook`, `verify`, `state diff`) and the 0.4 helpers
  (`ship` / `close` / `check --all`) are tooling ergonomics, not new protocol
  concepts.
